# 코드 검토 및 최적화 제안 (2026-04-28)

## 검토 범위
- 정적 구조/문법/전역/DOM 계약 검사 스크립트
- 대형 JS 모듈 중심 수동 리뷰

## 요약
- 현재 기본 게이트(`npm run check:web`)는 모두 통과했습니다.
- 단기적으로는 **렌더링/이벤트 처리 재구조화**, 중기적으로는 **데이터 조회 O(n) 반복 축소**, 장기적으로는 **대형 파일 분할**이 가장 효과적인 최적화 포인트입니다.

## 우선순위별 개선안

### P0 (바로 적용 권장)
1. **대형 렌더 함수의 이벤트 위임 일원화**
   - `buildStockMgmt()`는 매 렌더마다 HTML을 재생성하고, 이후 `_bindStockMgmtEvents()`에서 개별 핸들러를 다시 바인딩합니다.
   - 행 개수가 늘어날수록 재바인딩 비용이 누적되므로, 컨테이너 단일 `click` 위임으로 전환해 이벤트 등록 횟수를 상수화하는 것이 좋습니다.

2. **인라인 스타일 대량 세팅 최소화**
   - `views_history.js`의 `_renderHistBenchmarkButtons()`는 버튼마다 다수의 `style.*` 속성을 반복 갱신합니다.
   - 클래스토글(`.is-active`) + CSS 선언으로 이전하면 리플로우/리페인트 비용과 유지보수 비용을 같이 줄일 수 있습니다.

3. **네트워크 요청 상태 표준화**
   - `loadHistoryChart()` 내부에서 상태 문자열을 여러 분기에서 조합합니다.
   - 상태 메시지 렌더 함수(예: `setHistoryStatus(type, meta)`)를 분리하면 중복 문자열 생성 및 분기 누락을 줄일 수 있습니다.

### P1 (다음 스프린트 권장)
1. **종목 조회 인덱스 캐시 도입**
   - `data.js`의 `getEP`, `getEPByCode`는 `EDITABLE_PRICES.find(...)`를 반복 사용합니다.
   - 데이터 규모 증가 시 누적 O(n) 탐색 비용이 커지므로 `Map(name->item)`, `Map(code->item)` 캐시를 두고 저장 시 무효화하는 구조를 권장합니다.

2. **대형 파일 분할 (기능 경계 단위)**
   - `views_history.js`, `mgmt_editor.js`, `data.js`는 파일 길이가 길어 변경 충돌과 회귀 위험이 큽니다.
   - 차트 렌더, 테이블 렌더, API, 포맷터 유틸 등으로 모듈 분할하면 테스트 포인트가 명확해집니다.

3. **HTML 템플릿 조립 방식 개선**
   - 긴 템플릿 문자열 결합은 XSS 점검 포인트가 분산되고 diff 가독성이 떨어집니다.
   - 최소한 사용자 입력이 섞이는 위치는 escape 유틸 통일 적용을 권장합니다.

### P2 (구조 개선)
1. **전역 상태(window.*) 축소**
   - `window._histMode`, `window._histBenchmarks` 등 뷰 상태가 전역에 분산되어 있습니다.
   - 화면 단위 state 객체로 캡슐화하면 사이드이펙트 추적이 쉬워지고 테스트가 단순해집니다.

2. **비동기 요청 계층화**
   - GAS 제약으로 벤치마크 직렬 요청을 유지해야 하지만, 요청 큐/재시도 정책을 공통 레이어로 뽑으면 타 기능 재사용성이 높아집니다.
